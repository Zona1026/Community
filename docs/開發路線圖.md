# 開發路線圖

## 文件目的

本文件用來規劃「如何在 10 週內完成 Phase 1：結訓展示版」。

完整產品階段請以 `docs/產品路線圖.md` 為準。產品路線圖負責定義 Phase 1 到 Phase 6 的產品方向；開發路線圖只負責安排目前這段時間要怎麼推進。

這份週計畫是「預估安排」，不是固定死線。實際開發時可以依照進度調整，但調整時要遵守一個原則：

```text
Phase 優先順序高於週數安排
```

也就是說，週數只是用來估算節奏，不代表每週功能都不能變動。真正不能變的是：Phase 1 核心功能必須先完成，不能跳過 Phase 1 直接做 Phase 2 以後的功能。

## 核心目標

10 週內的主要目標仍然是穩定完成 Phase 1 結訓展示版，證明核心流程成立：

```text
資料整合
→ 熱門話題分類
→ AI 摘要分析
→ 爭議提醒
→ 內容靈感 / 文案生成
```

Phase 1 完成後，如果還有時間，可以依照 `docs/產品路線圖.md` 的順序，提前開始 Phase 2 中低風險、高展示價值的功能。

但不可以跳過 Phase 1 核心功能，直接挑戰 Phase 3、Phase 4、Phase 5 或 Phase 6 的功能。

## 開發原則

- 開發路線圖是預估安排，不是固定死線。
- Phase 優先順序高於週數安排。
- 10 週內主要目標是完成 Phase 1 結訓展示版。
- Phase 1 沒穩定完成前，不開發 Phase 2 以後功能。
- 如果 Phase 1 提前完成，可以依照產品路線圖順序提前開始 Phase 2。
- 提前加功能時，以低風險、高展示價值、不破壞 Demo 穩定性為原則。
- Demo 必須可以在 10 分鐘內完整跑完。
- 展示資料應可重複操作，不依賴臨場外部資料。

## Week 1~2：需求確認與架構設計

工作項目：

- 確認產品定位。
- 整理 docs 文件。
- 確認 Phase 1 範圍。
- 設計 mockData 格式。
- 設計 Normalize 後的標準資料格式。
- 規劃前端頁面。
- 規劃後端資料流程。

完成標準：

- 文件已清楚區分 Phase 1 與 Phase 2 以後功能。
- Phase 1 必做功能、暫不實作功能都已確認。
- mockData 欄位與 Normalize 後資料格式已定義。
- 前端主要頁面與後端資料流程已有初步規劃。
- 可以開始進入專案實作，不再反覆擴大需求範圍。

## Week 3~4：基礎專案與資料整合

工作項目：

- 建立 Vue 3 前端專案。
- 建立 Flask 後端專案。
- 建立 PostgreSQL 資料庫。
- 建立 mockData。
- 建立 JSON 匯入流程。
- 建立 Normalize 流程。
- 建立熱門分數計算。
- 完成 Threads / Dcard 假資料展示。

完成標準：

- 前端與後端專案可以正常啟動。
- 後端可以讀取 mockData 或 JSON 資料。
- Threads 與 Dcard 假資料可以被轉成統一格式。
- 系統可以計算基本熱門分數。
- 前端可以顯示整理後的假資料。

## Week 5~6：前端核心畫面

工作項目：

- 首頁 Dashboard。
- 三個核心區塊：
  1. 所有熱門話題
  2. 符合產業熱門話題
  3. 自訂關鍵字熱門話題
- 話題卡片。
- 話題詳情頁或詳情彈窗。
- 平台標籤。
- 熱門標籤。
- 爭議提醒顯示區。

完成標準：

- 首頁可以清楚展示熱門話題。
- 三個核心區塊都有資料或合理的空狀態。
- 使用者可以點擊話題卡片查看詳情。
- 話題詳情能呈現來源平台、熱門分數、內容摘要與相關標籤。
- 爭議提醒已有固定顯示位置，後續可接 AI 判斷結果。

## Week 7~8：AI 分析與內容生成

工作項目：

- AI 短摘要。
- AI 完整摘要。
- 爭議提醒判斷。
- 語氣選擇。
- 發文角度選擇。
- AI 內容靈感或文案生成。
- 最新三版文案保留。

完成標準：

- 單一話題可以產生 AI 短摘要。
- 單一話題可以產生 AI 完整摘要。
- 系統可以顯示爭議提醒判斷。
- 使用者可以選擇語氣與發文角度。
- 系統可以根據話題產生內容靈感或文案。
- 同一話題只保留最新三版生成結果。
- AI 失敗時有備援內容或錯誤提示，不影響 Demo 主流程。

## Week 9~10：整合、測試與 Demo 收斂

工作項目：

- 前後端 API 串接。
- 修正主要錯誤。
- 準備展示資料。
- 補強 UI 細節。
- 補強 README。
- 確認 Demo 流程。
- 確保不依賴臨場外部資料。

完成標準：

- Phase 1 Demo 主流程可以從頭跑到尾。
- 展示資料固定且可重複操作。
- README 能說明專案定位、啟動方式、展示流程與 Phase 1 範圍。
- 主要畫面沒有明顯破版或流程中斷問題。
- Demo 可以在 10 分鐘內穩定完成。

## Week 11：展示準備與緩衝週

工作項目：

- 製作簡報。
- 練習展示。
- 準備 Demo 講稿。
- 修正小問題。
- 若 Phase 1 已穩定完成，可開始 Phase 2 小功能。

完成標準：

- 展示流程、講稿與簡報已準備完成。
- Demo 操作順序已演練過。
- 小問題已修正或記錄為已知限制。
- 若有新增 Phase 2 小功能，不影響 Phase 1 Demo 穩定性。

## Week 12：結訓展示

工作項目：

- 最後確認展示環境。
- 進行結訓展示。
- 回答專案定位、資料整合、AI 分析與未來產品化問題。

完成標準：

- 展示環境可以正常啟動。
- 能清楚說明本專案的核心價值。
- 能說明目前完成的是 Phase 1 結訓展示版。
- 能說明 Phase 2 以後的產品化方向。

## 提前完成時的處理方式

如果 Phase 1 提前完成，不代表後續時間浪費。

此時應依照 `docs/產品路線圖.md` 的 Phase 順序，從 Phase 2 中挑選低風險、高展示價值的功能加入。

優先順序建議：

1. 文案一鍵複製
2. 平台篩選
3. 熱門標籤篩選
4. 手動重新匯入資料
5. 使用者設定儲存
6. 收藏熱門話題
7. 基本登入
8. JSON / CSV 真實資料匯入

不建議提前挑戰：

- Growth Rate
- Momentum
- Lifecycle Stage
- Cross Platform Signals
- AI 推薦觀察
- SaaS 權限系統
- 付費會員功能

原因：

這些功能屬於 Phase 3 以後，會明顯增加系統複雜度，容易影響結訓展示穩定性。

## Phase 1 最小可展示版本

- 匯入 mockData 或 JSON 資料。
- 將 Threads 與 Dcard 資料 Normalize 成統一格式。
- 計算熱門分數。
- 首頁顯示所有熱門話題。
- 首頁顯示符合產業熱門話題。
- 首頁顯示自訂關鍵字熱門話題。
- 顯示話題卡片。
- 顯示話題詳情頁或詳情彈窗。
- 產生 AI 短摘要。
- 產生 AI 完整摘要。
- 產生爭議提醒。
- 選擇語氣。
- 選擇發文角度。
- 產生內容靈感或文案。
- 保留同一話題最新三版文案。
- 完成基本使用者設定頁。

## 目前不應提前開發的內容

- 複雜趨勢生命週期。
- 跨平台爆紅判斷。
- 即將過氣判斷。
- AI 趨勢顧問聊天介面。
- 完整 SaaS 帳號權限。
- 付費會員與團隊功能。
- 自動寄送報告。
- 正式上線部署與長期維運功能。
