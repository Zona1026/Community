# 路線圖審視結論

Step 1 ~ Step 8 符合目前專案主要缺口，但原本「Step 1 直接移除 mock AI」風險過高。

調整後原則：

- 真實 AI 可以先開始，但必須保留 mock fallback
- PostgreSQL 與正式 API 要分階段落地
- 使用者設定應建立在資料庫與 API 基礎之後
- Source Analysis 仍可列為 Phase 2 後段，但不應早於第二資料來源
- Growth Rate、Lifecycle Stage、Trend Chart 先保留在 Phase 3 Lite

# Phase 1 提前完成後開發規劃

由於 Phase 1 已提前完成，因此後續開發依照產品路線圖順序推進。

優先補齊：

- 真實 AI
- 真實資料庫
- 正式 API
- 使用者體驗
- 第二資料來源

避免直接投入：

- SaaS 權限系統
- Workspace
- 團隊協作
- 訂閱系統
- AI 趨勢顧問

開發原則：

- 不破壞 Phase 1 Demo 穩定性
- 優先選擇低風險、高展示價值項目
- 每次只推進一個明確能力
- 每階段都保留 fallback，避免 Demo 因外部服務失敗而中斷
- 每個 Step 完成後都需通過 build、TypeScript 檢查與 Demo 主流程驗證

## 2026-06-09 Phase 3 Lite Backlog: AI Copy Editor Modal

### 功能名稱

AI Copy Editor Modal

### 需求摘要

目前 AI 生成文案後會直接顯示在話題詳情頁面區塊中。未來希望調整為：使用者點擊「生成文案」後，AI 完成生成並開啟 Modal，讓使用者可在 Modal 內檢視完整文案、直接編輯、使用一鍵複製，或關閉 Modal。

### Phase 3 Lite MVP 範圍

- Modal 顯示 AI 文案。
- Editable Textarea，可直接修改生成結果。
- Copy to Clipboard，一鍵複製目前 textarea 內容。
- Close Modal，關閉不影響既有 Dashboard 與話題詳情流程。

### 暫不實作

- 文案歷史紀錄。
- 多版本管理。
- AI 重新生成。
- 匯出功能。
- 自訂語氣。

### 可能影響檔案與元件

- `frontend/src/components/TopicDashboard.tsx`
  - 目前 `handleGenerateCopy()` 負責呼叫 `requestAiAnalysis()` 並將 `analysis.generatedCopies[0]` 寫入 `generatedCopiesByTopic`。

### 技術風險

- 低到中：功能主要是前端互動，不需新增 API 或資料庫。
- Clipboard API 風險：`navigator.clipboard` 需要瀏覽器支援與安全情境；需提供失敗提示或 fallback。
- Modal 疊層風險：目前已有 `TopicDetailModal`，新 Copy Editor Modal 若疊在其上方，需避免 backdrop 關閉錯誤層級。
- 狀態同步風險：使用者編輯後是否要回寫 `generatedCopiesByTopic` 需在實作前決定。MVP 建議不回寫歷史，只保留 Modal 內編輯文字。
- 可及性風險：需補 `role="dialog"`、`aria-modal="true"`、明確標題與關閉按鈕。Focus trap 可列為後續強化，MVP 至少確保鍵盤可操作。
