# 後端架構

## 技術選型

- Python Flask
- MySQL
- Flask-CORS
- MySQL Connector 或 SQLAlchemy
- APScheduler

第一版以單一 Flask 專案完成 API、資料匯入、資料標準化、AI 分析流程與文案生成，不拆成微服務。

本專案定位為「趨勢資料整合 + AI 分析 + 內容生成平台」。後端重點不是自行開發完整爬蟲，而是接收不同來源的資料，轉換成統一格式，並提供穩定的 AI 分析與文案生成能力。

## 後端目標

- 提供前端所需 API。
- 管理使用者設定。
- 匯入 mockData.json 或第三方工具產生的 JSON、CSV。
- 將不同平台資料標準化為統一格式。
- 整理熱門話題並儲存至 MySQL。
- 呼叫 AI 產生摘要、爭議提醒與文案。
- 控制每個話題只保留最新三版文案。

## 資料整合設計

第一版不自行開發完整爬蟲。原因如下：

- 開發成本高。
- 維護成本高。
- 容易受到平台改版影響。

系統資料來源分為三個階段。

### 開發階段

使用 `mockData.json` 作為固定資料來源。

用途：

- 先穩定完成首頁、詳細頁、AI 分析與文案生成。
- 避免開發期間被外部平台限制影響。
- 方便測試資料標準化流程。

### 展示階段

匯入第三方爬蟲工具產生的 JSON 或 CSV。

流程：

1. 使用第三方工具取得平台資料。
2. 匯出 JSON 或 CSV。
3. 後端匯入原始資料並存入 `raw_posts`。
4. 依平台轉換規則執行 Normalize。
5. 將標準化後資料存入 `posts`。
6. 依 `posts` 資料產生熱門話題、AI 摘要與文案。

### 進階版

未來可串接：

- Apify
- n8n
- 其他合法資料來源

進階版只需新增資料來源連接器與轉換器，不需重寫前端頁面與主要資料表。

## 資料標準化 Normalize

不同平台資料在欄位命名、互動數據與時間格式上都不一致，因此進入主要流程前必須轉換為統一格式。

標準欄位：

- platform_name
- platform_code
- external_post_id
- title
- body
- url
- likes
- comments
- shares
- published_at
- trend_score
- extra_data

### Normalize 流程

1. 匯入原始資料到 `raw_posts`。
2. 判斷資料來源平台，例如 Threads、Dcard。
3. 使用對應平台的轉換規則。
4. 將原始欄位轉換為標準欄位。
5. 計算 `trend_score`。
6. 將無法對應但仍有參考價值的資料放入 `extra_data`。
7. 儲存至 `posts`。

未來若新增 PTT、Instagram、Facebook 等平台，只需新增對應平台的資料轉換流程，不需要修改前端頁面與主要資料表。

## API 規劃

### 使用者設定

#### 取得設定

```text
GET /api/settings
```

回傳：

- 產業類別
- 常用語氣
- 每天資料匯入時間
- 自訂關鍵字
- 是否完成首次設定

#### 更新設定

```text
PUT /api/settings
```

用途：

- 儲存首次設定。
- 修改設定頁資料。

### 熱門話題

#### 取得首頁話題

```text
GET /api/topics/home
```

回傳三個區塊：

- 所有熱門話題
- 符合產業熱門話題
- 自訂關鍵字熱門話題

#### 取得單一話題詳情

```text
GET /api/topics/{post_id}
```

回傳：

- 原始內容
- 熱門資料
- AI 短摘要
- AI 完整摘要
- 爭議提醒
- 最新三版文案

#### 手動匯入資料

```text
POST /api/import/posts
```

用途：

- 匯入 `mockData.json`。
- 匯入第三方工具產生的 JSON 或 CSV。
- 模擬每日排程匯入效果。

### 語氣管理

#### 取得語氣清單

```text
GET /api/tones
```

包含預設語氣與使用者自訂語氣。

#### 新增自訂語氣

```text
POST /api/tones
```

欄位：

- 語氣名稱
- 語氣描述

#### 修改自訂語氣

```text
PUT /api/tones/{tone_id}
```

#### 刪除自訂語氣

```text
DELETE /api/tones/{tone_id}
```

### AI 文案

#### 產生文案

```text
POST /api/topics/{post_id}/copies
```

欄位：

- 語氣
- 發文角度

流程：

1. 取得話題內容。
2. 取得使用者選擇的語氣與發文角度。
3. 呼叫 AI 產生社群文案。
4. 儲存文案版本。
5. 若超過三版，刪除最舊版本。
6. 回傳最新三版文案。

## 資料匯入流程

### 排程匯入

使用 APScheduler 依使用者設定時間每天執行。

第一版若多使用者處理過於複雜，可先以單一展示使用者為主。

流程：

1. 讀取使用者設定的匯入時間。
2. 讀取 `mockData.json` 或指定的 JSON、CSV 檔案。
3. 將原始資料存入 `raw_posts`。
4. 執行 Normalize，產生 `posts` 標準化資料。
5. 計算熱門分數。
6. 產生熱門標籤。
7. 呼叫 AI 產生摘要與爭議提醒。
8. 將分析結果存入資料庫。

### 手動匯入

展示時可提供手動按鈕呼叫後端 API，方便老師或同學立即看到流程。

## 平台資料轉換規劃

### Threads

第一版使用 `mockData.json` 或第三方工具產生的 JSON、CSV。若正式 API 或資料取得限制過高，可在文件與展示中說明限制。

可能原始欄位：

- 標題或貼文摘要
- 原始內容
- 原始連結
- 按讚數
- 留言數
- 發文時間

### Dcard

第一版可使用 `mockData.json` 或第三方工具整理後的資料作為展示來源。

可能原始欄位：

- 標題
- 文章內容
- 文章連結
- 愛心數
- 回覆數
- 發文時間

## AI 分析流程

### 短摘要

輸入：

- 標題
- 原始內容

輸出：

- 一句話短摘要

### 完整摘要

輸入：

- 標題
- 原始內容
- 留言或互動數據

輸出：

- 事件背景
- 大家在意什麼
- 適合怎麼發文

### 爭議提醒

判斷條件：

- 食安問題
- 政治議題
- 名人爭議
- 社會事件

輸出：

- 是否有爭議風險
- 爭議原因
- 提醒文字

## 熱門標籤產生規則

第一版可用簡單規則產生。

- 留言數最高：留言數位於前幾名
- 按讚數最高：按讚數位於前幾名
- 最多人討論：留言數高於設定門檻

## 錯誤處理

### API 錯誤

回傳一致格式：

```json
{
  "success": false,
  "message": "錯誤訊息"
}
```

### AI 失敗

若 AI 呼叫失敗：

- 保留原始話題資料。
- 摘要欄位顯示尚未產生。
- 前端可提供重新分析按鈕作為加分功能。

### 匯入或轉換失敗

若單一平台匯入或轉換失敗：

- 記錄錯誤。
- 不影響其他平台資料。
- 前端顯示最後成功更新時間。

## 安全與設定

敏感設定放在環境變數：

- MySQL 連線資訊
- AI API Key
- Flask Secret Key

第一版可先不做完整登入系統，以單一使用者設定為主。若時間允許，再加入簡單帳號登入。

## 驗收重點

- API 可正常供前端取得設定與熱門話題。
- 可手動匯入 mockData.json 或 JSON、CSV。
- 可依排程時間執行資料匯入流程。
- 可將原始資料轉換為標準化資料。
- AI 摘要與文案可正確儲存。
- 每個話題只保留最新三版文案。
- 發生匯入、轉換或 AI 錯誤時，系統不會整體中斷。
