環境變數：

- `AI_PROVIDER=mock`：預設使用 mock / rule-based AI
- `AI_PROVIDER=openai`：嘗試使用 OpenAI adapter
- `OPENAI_API_KEY`：OpenAI API Key
- `OPENAI_MODEL`：OpenAI model，預設 `gpt-4o-mini`

fallback 規則：

- 未設定 `AI_PROVIDER=openai` 時使用 mock fallback
- 未設定 `OPENAI_API_KEY` 時使用 mock fallback
- OpenAI API 回傳錯誤時使用 mock fallback
- 前端呼叫 `/api/ai-analysis` 失敗時使用本地 mock fallback

安全原則：

- 不移除 mock / rule-based AI
- 不在 client component 暴露 API Key
- Demo 未設定 OpenAI API Key 時仍可正常展示
- 真實 AI 失敗不影響話題詳情與文案生成流程

## 2026-06-11 Ingestion Safety Boundary MVP

### 目標

- `.env.example` 新增 `ALLOW_DEV_ENDPOINTS=false`。
- production 環境應保持 `ALLOW_DEV_ENDPOINTS=false` 或不設定。
- `/dev/*` 不視為 production ingestion control surface。
- `/api/topics`、`/api/user-settings`、`/api/favorites` 不受 guard 影響。

### Smoke Batch Metadata 規格

不新增資料表，沿用 `topics`：

- `topics.source`
  - `rss_smoke_test`
  - `podcast_smoke_test`
  - `dcard_smoke_test`
- `topics.topic_key` prefix
  - `rss-smoke-`
  - `podcast-smoke-`
  - `dcard-smoke-`
- `topics.payload.smokeBatchId`
  - 對應每次 smoke write 產生的 `batch_id`
- source-specific metadata
  - RSS: `payload.rssSmokeTest` / `payload.rssIngestion`
  - Podcast: `payload.podcastSmokeTest`
  - Dcard: `payload.dcardSmokeTest`

### Rollback Safety

RSS / Podcast / Dcard rollback 均需同時符合以下條件才刪除：

- `source` 等於對應 smoke source
- `payload ->> 'smokeBatchId'` 等於指定 `batch_id`
- `topic_key` 符合對應 smoke prefix

這可避免 rollback 誤刪非 smoke topics。

- 無 schema 變更，無 production scheduler 變更。

## 2026-06-11 Per-source PostgreSQL Advisory Lock

### 目標

在不新增 scheduler、不設定 cron、不新增 UI 的前提下，讓既有 Manual Scheduled Ingestion Runner 具備「同一 source 不可同時執行兩次」的保護。

### Lock Key 規則

Runner 會以以下字串建立 deterministic lock name：

```text
scheduled-ingestion:{source_type}:{source_key}
```

再用 SHA-256 轉成 PostgreSQL advisory lock 需要的 signed 64-bit integer key。這讓：

- 同一個 `source_type + source_key` 使用同一把 lock。
- 不同 source 使用不同 lock，可各自執行。

### Lock Unavailable 行為

如果 `pg_try_advisory_lock` 取不到 lock：

- 不執行 fetch。
- 不執行 parse。
- 不執行 topic draft。
- 不執行 topics upsert。
- 寫入一筆 `ingestion_runs`。
- `status=skipped`。
- `error_count=1`。
- `error_message=Scheduled ingestion lock unavailable; another run is already active for this source.`
- `metadata.advisoryLock.reason=lock_unavailable_already_running`。
- CLI exit code = 1，因為 CLI 只有 `status=success` 才回 0。

### Lock Release

Advisory lock 使用 session-level `pg_try_advisory_lock`，runner 會持有一條 lock connection，並在 `finally` 中呼叫 `pg_advisory_unlock` 後關閉 connection。正常成功、失敗、blocked 或 skipped 都會釋放 lock，不應永久卡住。

### 邊界

- 這是 production scheduler 前的 runner 保護。
- 不是 scheduler。
- 沒有新增資料表。
- 沒有新增環境變數。
- 沒有新增 UI。

## 2026-06-12 Render psycopg Binary Build Fix

### 問題

Render backend 第一次部署失敗：

```text
No matching distribution found for psycopg-binary==3.2.3
```

原本 `backend/requirements.txt` 使用：

```text
psycopg[binary]==3.2.3
```

`psycopg[binary]` 是 Psycopg 3 的 binary extra，pip 會解析並安裝同版本的 `psycopg-binary`。因此 Render 實際尋找的是 `psycopg-binary==3.2.3`。

### 原因

本機可安裝，是因為目前本機 Python / Windows / Conda 環境已有相容 wheel 或已安裝該版本。Render Linux build 使用不同 Python/runtime/platform，`psycopg-binary==3.2.3` 沒有符合該環境的 distribution，因此 build 失敗。

### 修正

維持 Psycopg 3，不切換到 `psycopg2-binary`，因為 backend 程式碼使用：

```text
import psycopg
```

將 dependency 升級到同系列較新的版本：

```text
psycopg[binary]==3.2.13
```

### Render 重新部署前

- 確認 `backend/requirements.txt` 已更新。
- commit 並 push 到 Render 連接的 repo/branch。
- 重新觸發 Render backend deploy。
- 不需要修改 Render Start Command，仍使用 `gunicorn app:app`。
