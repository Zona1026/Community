# 資料庫設計

本文件描述資料庫設計方向與 schema 候選，不維護目前狀態。

目前專案狀態請以 `docs/CURRENT_STATUS.md` 為準。技術選型請以 `docs/技術資源與選型.md` 為準。

## 設計原則

- 使用 PostgreSQL 作為正式資料庫。
- 使用 `DATABASE_URL` 管理連線。
- 不硬編碼帳密或供應商資訊。
- 外部資料庫不可用時，Demo 應保留 fallback。
- 未來若需要 embedding 或向量搜尋，再評估 pgvector。

## 2026-06-10 Topics Read MVP Schema Note

本次為 PostgreSQL Topics Read MVP 做最小 schema 更新。

### topics

目前 `topics` table 保留原有摘要欄位，並新增：

- `payload JSONB NOT NULL DEFAULT '{}'::jsonb`

用途：

- 保存完整 DashboardTopic / TrendSignal-compatible payload。
- 支援前端 topic detail 需要的欄位，例如 `growthRate`、`momentum`、`lifecycleStage`、`scoreHistory`、`relatedContent`、`inspirationIdeas`。
- 避免在 MVP 階段拆出多張複雜 trend report / history tables。

目前讀取策略：

```text
Flask /api/topics
-> PostgreSQL topics rows
-> row.payload normalized to frontend topic shape
-> no rows / no DB / DB failure fallback to backend/mock_topics.py
```

Seed 策略：

- `backend/mock_topics.py` 是目前 seed/mock records 的來源。
- `/db/smoke-test` 在 `DATABASE_URL` 可用時會 upsert mock topics 到 `topics.payload`。

限制：

- 本次不新增正式寫入 API。
- 本次不新增爬蟲、外部 API、排程或資料清洗 pipeline。
- 未來若進入正式資料 ingestion，應再評估是否拆出 `content_items`、`trend_reports`、`score_history` 等正規化 tables。

## 2026-06-10 Reddit Posts Staging Table

本次為 Reddit Ingestion MVP Step 1 新增 staging table：

```text
reddit_posts
```

用途：

- 保存 Reddit API ingestion 的原始 post 資料。
- 作為未來轉換成 `topics.payload` / TrendSignal-compatible data 的 staging layer。
- 避免外部來源資料直接污染正式 topic payload。

### 欄位

- `id`
- `subreddit`
- `reddit_post_id`
- `title`
- `url`
- `author`
- `score`
- `num_comments`
- `created_utc`
- `fetched_at`
- `raw_json`

### Constraint / Index

- `reddit_post_id` 為 unique，作為未來 upsert key。
- `idx_reddit_posts_subreddit` 用於依 subreddit 查詢。
- `idx_reddit_posts_created_utc` 用於依 Reddit 建立時間排序。

### 與 topics / TrendSignal 的映射方向

- `reddit_post_id` -> topic external id / payload id。
- `title` -> topic。
- `score` + `num_comments` -> TrendSignal score 的最小來源。
- `subreddit` -> source / platformTags。
- `created_utc` -> freshness / lifecycle 的未來計算來源。
- `raw_json` -> 保留完整 Reddit API response 供後續資料轉換使用。

本次不實作 Reddit API client、OAuth、ingestion script、排程或 topics 自動生成。

## 2026-06-10 RSS Normalized Dry-run Shape

本次沒有修改 `rss_items` schema，只新增 read-only dry-run mapper，確認 RSS item 在寫入前可對齊既有 table 欄位。

### Normalized 欄位對照

- `source_name` -> RSS channel title / Atom feed title。
- `feed_url` -> `RSS_FEED_URL`。
- `item_guid` -> RSS `guid` / Atom `id`；若缺失則 fallback 使用 `link`。
- `title` -> RSS / Atom item title，缺失時 skipped。
- `link` -> RSS / Atom item link，缺失時 skipped。
- `author` -> RSS `author`、`dc:creator` 或 Atom author name。
- `published_at` -> RSS `pubDate`、Atom `published` 或 `updated`，轉為 UTC ISO datetime；無法解析時為 `null`。
- `fetched_at` -> dry-run 執行時間 UTC ISO datetime。
- `summary` -> RSS `description`、Atom `summary` 或 `content`。
- `raw_json` -> 保留原始解析欄位與 `guid_was_fallback` 標記。

### Dry-run 結論

- `total_parsed_items=30`
- `valid_normalized_items=30`
- `skipped_items_count=0`
- `missing_guid_fallback_count=30`
- `writes_to_database=False`

目前 normalized shape 已可支援下一步 `rss_items` insert/upsert；下一步需補實際 DB 寫入與 unique `item_guid` upsert 驗證。

## 2026-06-11 Topics Upsert Smoke Schema Confirmation

本次確認 `topics.topic_key` 已可作為 topics upsert 的唯一鍵：

- `topics.topic_key` 欄位存在。
- `topics.topic_key` 具備 single-column unique constraint。
- `POST /dev/topics-upsert-smoke` 使用 `ON CONFLICT (topic_key) DO UPDATE` 驗證 insert/update。
- 第一次 smoke 呼叫 insert，第二次 smoke 呼叫 update。
- 同一個 `topic_key=topics-upsert-smoke-test` 最終只保留一筆 topic，未產生重複資料。

本次沒有修改資料庫 schema。若未來環境缺少此支援，最小修正方向是：

```sql
ALTER TABLE topics ADD COLUMN IF NOT EXISTS topic_key TEXT;
CREATE UNIQUE INDEX IF NOT EXISTS topics_topic_key_key ON topics(topic_key);
```

## 2026-06-11 RSS-to-Topics Ingestion Schema Usage

本次確認 manual RSS-to-topics ingestion 可使用既有 `topics` schema：

- `topics.topic_key` 作為 RSS grouped topic 的 upsert key。
- `topics.title` 儲存 grouped payload representative title。
- `topics.summary` 儲存 grouped payload summary。

- `topics.payload` 儲存 `/api/topics` 可讀的 topic payload，包括 `id`、`topic`、`summary`、`source`、`relatedContent`、`platformTags`、`topicTags` 與 `rssIngestion` evidence metadata。

本次沒有修改資料庫 schema。後續若要支援 production ingestion audit trail，可再評估是否新增獨立的 topic evidence / source mapping table。

## 2026-06-11 RSS-generated Topics Status Schema Note

本次新增 read-only RSS-to-topics status endpoint，沒有修改資料庫 schema。

目前用來判斷 RSS-generated topics 的既有欄位：

- `topics.payload.rssIngestion`

目前可讀欄位來源：

- `id` -> `topics.id`
- `topic_key` -> `topics.topic_key`
- `title` -> `topics.title`
- `source_type` -> `topics.payload.rssIngestion.sourceType`
- `source_name` -> `topics.payload.rssIngestion.sourceName`
- `source_url` -> `topics.payload.rssIngestion.sourceUrl`
- `evidence_count` -> `topics.payload.rssIngestion.evidenceCount`
- `created_at` -> `topics.created_at`
- `updated_at` -> `topics.updated_at`

Schema 缺口：

- 目前沒有 dedicated `topics.source_type` 欄位。
- 目前沒有獨立的 `topic_sources` 或 `topic_evidence` table。
- MVP 階段使用 `topics.payload.rssIngestion` 足以安全判斷由本 ingestion flow 產生的 topics。

後續最小 schema 方向可考慮：

```sql
CREATE TABLE topic_evidence (
    id BIGSERIAL PRIMARY KEY,
    topic_id BIGINT NOT NULL REFERENCES topics(id) ON DELETE CASCADE,
    source_type TEXT NOT NULL,
    source_item_id TEXT NOT NULL,
    source_url TEXT NOT NULL DEFAULT '',
    evidence_title TEXT NOT NULL DEFAULT '',
    evidence_summary TEXT NOT NULL DEFAULT '',
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

## 2026-06-11 RSS Smoke Test Batch Markers

本次低風險 RSS 真實資料 smoke flow 沒有修改 schema，使用既有 `topics` 欄位標記 smoke 寫入：

- `topics.source = 'rss_smoke_test'`
- `topics.topic_key` 使用 `rss-smoke-` 前綴
- `topics.payload.smokeBatchId` 儲存每次寫入產生的 `batch_id`
- `topics.payload.rssSmokeTest` 儲存 smoke metadata

Rollback endpoint 使用以下條件刪除 smoke 寫入資料：

```sql
DELETE FROM topics
WHERE source = 'rss_smoke_test'
AND payload ->> 'smokeBatchId' = :batch_id;
```

此設計避免影響 production topics，也讓 smoke batch 可以用 `batch_id` 回滾。
