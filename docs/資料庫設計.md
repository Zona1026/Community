# 資料庫設計

## 資料庫

資料庫使用 MySQL。

第一版以單一使用者展示為主，但資料表保留 `user_id` 欄位，方便未來擴充多使用者。

## 資料表總覽

- users
- user_settings
- user_keywords
- tones
- raw_posts
- posts
- topic_tags
- topic_ai_analysis
- copy_versions
- import_logs

## users

使用者資料表。

| 欄位 | 型別 | 說明 |
| --- | --- | --- |
| id | INT PK AUTO_INCREMENT | 使用者 ID |
| name | VARCHAR(100) | 使用者名稱 |
| email | VARCHAR(255) | Email |
| created_at | DATETIME | 建立時間 |
| updated_at | DATETIME | 更新時間 |

第一版若不做登入，可建立一筆預設使用者資料。

## user_settings

使用者設定資料表。

| 欄位 | 型別 | 說明 |
| --- | --- | --- |
| id | INT PK AUTO_INCREMENT | 設定 ID |
| user_id | INT | 使用者 ID |
| industry_category | VARCHAR(100) | 產業類別 |
| default_tone | VARCHAR(100) | 常用語氣 |
| daily_import_time | TIME | 每天資料匯入時間 |
| is_setup_completed | BOOLEAN | 是否完成首次設定 |
| created_at | DATETIME | 建立時間 |
| updated_at | DATETIME | 更新時間 |

## user_keywords

使用者自訂關鍵字資料表。

| 欄位 | 型別 | 說明 |
| --- | --- | --- |
| id | INT PK AUTO_INCREMENT | 關鍵字 ID |
| user_id | INT | 使用者 ID |
| keyword | VARCHAR(100) | 自訂關鍵字 |
| created_at | DATETIME | 建立時間 |

範例：

- AI
- 雅方
- iPhone
- 房價

## tones

語氣資料表。

| 欄位 | 型別 | 說明 |
| --- | --- | --- |
| id | INT PK AUTO_INCREMENT | 語氣 ID |
| user_id | INT NULL | 使用者 ID，預設語氣可為 NULL |
| name | VARCHAR(100) | 語氣名稱 |
| description | TEXT | 語氣描述 |
| is_default | BOOLEAN | 是否為系統預設語氣 |
| created_at | DATETIME | 建立時間 |
| updated_at | DATETIME | 更新時間 |

預設資料：

| name | description |
| --- | --- |
| 激動 | 用高情緒、強烈感受的方式撰寫 |
| 疑惑 | 用提問、好奇、不確定感的方式撰寫 |
| 專業 | 用理性分析、可信任的方式撰寫 |
| 幽默 | 用輕鬆有趣的方式撰寫 |
| 溫暖 | 用親切、理解、陪伴感的方式撰寫 |

自訂範例：

| name | description |
| --- | --- |
| 酸民風 | 用吐槽但不失禮貌的方式撰寫 |

## raw_posts

原始貼文資料表，用來保存從 `mockData.json`、JSON、CSV 或第三方工具匯入的原始資料。

此表保留資料來源的原始樣貌，方便 Debug、重新 Normalize 或未來調整轉換規則。

| 欄位 | 型別 | 說明 |
| --- | --- | --- |
| id | INT PK AUTO_INCREMENT | 原始資料 ID |
| source_name | VARCHAR(100) | 資料來源名稱，例如 mockData、Apify、n8n |
| platform_name | VARCHAR(100) | 平台名稱，例如 Threads、Dcard |
| platform_code | VARCHAR(50) | 平台代碼，例如 threads、dcard |
| external_post_id | VARCHAR(255) | 平台原始貼文 ID |
| raw_data | JSON | 原始資料完整內容 |
| imported_at | DATETIME | 匯入時間 |
| normalized_at | DATETIME NULL | 轉換完成時間 |
| status | VARCHAR(50) | 狀態，例如 pending、normalized、failed |
| error_message | TEXT NULL | 轉換失敗原因 |

## posts

標準化貼文資料表，用來存放 Normalize 後的統一格式資料。

前端首頁、話題詳細頁、AI 摘要、爭議提醒與文案生成都以 `posts` 為主要資料來源，不直接依賴不同平台的原始欄位。

| 欄位 | 型別 | 說明 |
| --- | --- | --- |
| id | INT PK AUTO_INCREMENT | 貼文 ID |
| raw_post_id | INT | 對應 raw_posts.id |
| platform_name | VARCHAR(100) | 平台名稱，例如 Threads、Dcard |
| platform_code | VARCHAR(50) | 平台代碼，例如 threads、dcard |
| external_post_id | VARCHAR(255) | 平台原始貼文 ID |
| title | VARCHAR(255) | 標題 |
| body | TEXT | 內文 |
| url | VARCHAR(500) | 原始連結 |
| likes | INT | 按讚數 |
| comments | INT | 留言數 |
| shares | INT | 分享數 |
| published_at | DATETIME | 原文發布時間 |
| trend_score | INT | 熱門分數 |
| extra_data | JSON | 額外資料 |
| created_at | DATETIME | 建立時間 |
| updated_at | DATETIME | 更新時間 |

熱門分數：

```text
trend_score = likes + comments * 2 + shares
```

### raw_posts 與 posts 差異

| 資料表 | 用途 | 說明 |
| --- | --- | --- |
| raw_posts | 保存原始資料 | 保留第三方工具或 mockData.json 的原始欄位，方便追蹤與重新轉換 |
| posts | 保存標準化資料 | 提供前端、AI 分析與文案生成使用，欄位固定且跨平台一致 |

不同平台資料進入系統前，必須先轉換為 `posts` 的統一格式。

標準欄位：

- platform_name
- platform_code
- external_post_id
- title
- body
- url
- likes
- comments
- shares
- published_at
- trend_score
- extra_data

未來若新增 PTT、Instagram、Facebook 等平台，只需新增資料轉換流程，不需要修改前端頁面與主要資料表。

## topics

熱門話題概念說明。

第一版可直接以 `posts` 作為熱門話題主表，不一定需要額外建立 `topics` 資料表。前端顯示的「熱門話題」即為標準化後且依 `trend_score` 排序的 `posts`。

若未來需要將多篇貼文合併成同一個話題，再另外新增 `topics` 彙整表。

### 舊版 topics 欄位對應

| 舊欄位 | 新欄位 |
| --- | --- |
| platform | posts.platform_name |
| title | posts.title |
| original_content | posts.body |
| original_url | posts.url |
| like_count | posts.likes |
| comment_count | posts.comments |
| hot_score | posts.trend_score |
| posted_at | posts.published_at |

## topic_tags

熱門標籤資料表。

| 欄位 | 型別 | 說明 |
| --- | --- | --- |
| id | INT PK AUTO_INCREMENT | 標籤 ID |
| post_id | INT | 貼文 ID，對應 posts.id |
| tag_name | VARCHAR(100) | 標籤名稱 |
| created_at | DATETIME | 建立時間 |

標籤範例：

- 最多人討論
- 按讚數最高
- 留言數最多

## topic_ai_analysis

AI 分析結果資料表。

| 欄位 | 型別 | 說明 |
| --- | --- | --- |
| id | INT PK AUTO_INCREMENT | 分析 ID |
| post_id | INT | 貼文 ID，對應 posts.id |
| short_summary | VARCHAR(255) | 一句話短摘要 |
| full_summary | TEXT | 完整摘要 |
| is_controversial | BOOLEAN | 是否涉及爭議 |
| controversy_reason | TEXT | 爭議原因 |
| controversy_warning | VARCHAR(255) | 爭議提醒文字 |
| created_at | DATETIME | 建立時間 |
| updated_at | DATETIME | 更新時間 |

爭議提醒文字：

```text
此話題可能涉及爭議，建議謹慎使用
```

## copy_versions

AI 文案版本資料表。

| 欄位 | 型別 | 說明 |
| --- | --- | --- |
| id | INT PK AUTO_INCREMENT | 文案 ID |
| post_id | INT | 貼文 ID，對應 posts.id |
| user_id | INT | 使用者 ID |
| version_number | INT | 版本編號 |
| tone_name | VARCHAR(100) | 語氣名稱 |
| angle | VARCHAR(100) | 發文角度 |
| content | TEXT | 文案內容 |
| created_at | DATETIME | 建立時間 |

發文角度：

- 跟風討論
- 專業分析
- 品牌觀點
- 幽默迷因

版本規則：

- 每個 `post_id` 加 `user_id` 最多保留三筆。
- 新增第四筆時，刪除最舊的一筆。
- `version_number` 可持續遞增，不需要重排。

## import_logs

資料匯入紀錄表。

| 欄位 | 型別 | 說明 |
| --- | --- | --- |
| id | INT PK AUTO_INCREMENT | 紀錄 ID |
| platform | VARCHAR(50) | 平台名稱 |
| status | VARCHAR(50) | 狀態，例如 success、failed |
| message | TEXT | 訊息或錯誤內容 |
| imported_count | INT | 匯入筆數 |
| normalized_count | INT | 成功標準化筆數 |
| started_at | DATETIME | 開始時間 |
| finished_at | DATETIME | 結束時間 |

用途：

- 顯示最後更新時間。
- Debug 匯入與 Normalize 流程。
- 展示排程匯入執行紀錄。

## 第一版簡化建議

若開發時間不足，可簡化：

- 不做 users 登入，只保留一位預設使用者。
- topic_tags 可先用一張表儲存，也可先在 posts 加上簡單文字欄位。
- import_logs 可先只記錄成功與失敗，不做詳細監控。
- AI 分析結果可在匯入完成後批次產生，不需要即時產生。
